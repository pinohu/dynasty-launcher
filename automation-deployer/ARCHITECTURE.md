## Core objects

### 1. Automation (in `registry/automations.json`)

One of 353 atomic units of work. Shape:

```json
{
  "id": "1.01",
  "slug": "gbp-monitor",
  "title": "Google Business Profile Monitor",
  "category_id": 1,
  "category": "Lead Generation & Prospecting",
  "task": "Monitor a GBP listing for new reviews, Q&A, and ranking changes daily.",
  "trigger": "cron",
  "trigger_detail": "daily",
  "output": "Slack/email alert with changes detected",
  "stack": ["n8n", "google_business_api", "emailit"],
  "personas": ["solo_steve", "manager_maria"],
  "jtbd": ["J19", "J21"],
  "service_blueprint": ["SB05"],
  "journey_stages": ["D4"],
  "manifest_ref": "1.01-gbp-monitor.yaml",
  "tier_hint": "foundation"
}
```

### 2. Manifest (in `manifests/<id>-<slug>.yaml`)

Declarative deployment spec. This is the contract the provisioner executes.

```yaml
id: "1.01"
slug: gbp-monitor
version: 1
artifacts:
  n8n_workflow: workflows/1.01-gbp-monitor.json
  code: null  # or ref to code module if not purely n8n
credentials_required:
  - key: GOOGLE_BUSINESS_API_KEY
    source: tenant   # tenant | deployer | customer-input
    scope: google_business.readonly
  - key: EMAILIT_API_KEY
    source: tenant
webhooks:
  inbound: []
  outbound:
    - name: alert_channel
      default: "{{tenant.comms.primary_email}}"
env_schema:
  GBP_LOCATION_ID:
    required: true
    prompt: "Your Google Business Profile location ID"
  ALERT_THRESHOLD:
    required: false
    default: "new_review"
    enum: ["new_review", "rating_drop", "qa_new", "ranking_change"]
triggers:
  - type: cron
    expression: "0 7 * * *"
depends_on: []
health_check:
  type: n8n_execution
  expect_within: 25h
rollback:
  - deactivate_workflow
  - delete_workflow
estimated_deploy_seconds: 35
```

### 3. Tenant (in `tenants/<slug>/`)

One business's deployment context.

```yaml
# tenants/acme-plumbing/tenant.yaml
slug: acme-plumbing
business_name: Acme Plumbing LLC
industry: field_service
vertical: plumbing
size: small_team          # solo | small_team | growth | franchise | agency
revenue_band: 500k-2m
primary_persona: manager_maria
secondary_personas: [field_service_fred]
infra:
  github:
    owner: acme-plumbing
    default_repo: acme-automations
  vercel:
    team_id: team_xxx
    project_id_pattern: acme-{automation_slug}
  n8n:
    base_url: https://n8n.acmeplumbing.com
    auth_ref: n8n_jwt             # references secrets.env
  domain: acmeplumbing.com

  - never_miss_a_lead
  - standardize_team_processes
pains:
  - leads_slip_through_cracks
  - team_forgets_steps
selected_automations: []          # filled by `interview`
```

### 4. Plan (generated by `deployer plan`)

The ordered, credential-complete, dependency-resolved deployment plan.

## Phases

### Phase A — Interview (`deployer interview`)

Input: tenant.yaml (empty selections) + optional LLM-assisted Q&A.

Output: `tenants/<slug>/selected-automations.yaml` — a scored list of automations ranked by **fit** against the tenant's persona, pains, and goals.

Selection sources:
1. **Hard rules** — `registry/selection-rules.json` (e.g., `if persona == field_service_fred → include 38.*`).
2. **Soft match** — match tenant `pains` against automation pain tags in the UX Bible.
3. **Manual override** — operator edits the YAML.

### Phase B — Plan (`deployer plan`)

1. Load each selected manifest.
2. Resolve `depends_on` (topological sort).
3. Compute the **credential gap**: which secrets are missing on the tenant.
4. Emit a `tenants/<slug>/plan.json` and a human-readable summary.

No side effects.

### Phase C — Provision (`deployer deploy`)

For each automation in dependency order, run its driver chain:

| Driver | Responsibility |
|---|---|
| `n8n` | Import/activate workflow, set credentials, register webhook URL |
| `vercel` | Create project (if needed), set env vars, trigger deploy |
| `github` | Create repo, push scaffold, add secrets to Actions if needed |
| `stripe` | Create products, prices, webhook endpoints |
| `clerk` / `auth` | Register allowed redirect URLs |
| `dns` (20i / CF / Route53) | Verify domain, set SPF/DKIM/DMARC if needed |
| `webhook-router` | Register inbound webhook with HMAC secret |
| `manual` | Emit step into `MANUAL-ACTIONS.md` for the tenant to execute |

Each driver is idempotent. Re-running `deploy` on a deployed automation is a no-op.

### Phase D — Verify (`deployer verify`)

- Poll each automation's `health_check`.
- Ping registered webhooks with a synthetic payload.
- Produce `tenants/<slug>/status.json`.

### Phase E — Rollback (`deployer rollback`)

- Per manifest, run the `rollback:` step list in reverse order.
- Leaves audit log in `tenants/<slug>/history/`.

## Credential boundary

Enforced end-to-end to match `CLAUDE.md`:

- The deployer **does not** store Stripe live secrets or webhook secrets for the tenant's customers on deployer infrastructure — only **routing metadata** (URL paths, event type filters).
- Tenant secrets live in `tenants/<slug>/.secrets/` which is **gitignored** and **encrypted at rest** (age / sops). A deployment run loads them, pushes them to the tenant's Vercel/n8n, and forgets them.
- The `source: deployer` credential type is allowed only for one-time provisioning (creating resources on the tenant's account via OAuth app or limited-scope PAT) — never for long-running service traffic.
- Everything that lands in the tenant's stack must be runnable **without** the deployer being online.

## Observability

- Every deployment run emits structured JSON events to `tenants/<slug>/history/<ts>.jsonl`.
- Each automation publishes a heartbeat back to the deployer's optional `api/webhook-router.js` for aggregated health — this is **opt-in** per tenant and carries no business payload.

## Extension points

- Add a new automation → write `manifests/<id>-<slug>.yaml` + (optional) `workflows/<id>-<slug>.json` + (optional) `registry/stacks.json` entry.
- Add a new driver → drop a module under `deployer/lib/drivers/` exporting `{ provision, verify, rollback }`.
- Add a new tenant type/template → `tenants/_template/variants/`.

See `DEPLOYMENT_MODEL.md` for per-topology deployment details.
