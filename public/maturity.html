What ships in an autonomous business unit

This page exists for one reason: accountability before purchase. Dynasty now targets a business-complete operating unit: website, funnel, products, CRM, RevOps, payments, onboarding, support, PostHog-ready analytics, AI agents, MCP tools, open-weight model routing, optional video/voice assets, repair checkpoints, workflows, and self-hosted infrastructure, with credential-gated integrations called out plainly.

Items marked "Planned" are not implied in default Enterprise unless listed in your tier's TIER_MODULES row; most require Custom Volume and configured keys. Credential boundary: launcher-held keys (e.g. DYNASTY_TOOL_CONFIG) are for generating deliverables and orchestrating setup; sustained operation of deployed apps is intended to use customer-owned vendor credentials in the shipped project (.env.example, MANUAL-ACTIONS, their Vercel env) — not an ongoing dependency on the launcher’s shared API pool. Customer Vercel projects are not auto-filled with real Clerk/Stripe/DB secrets from the launcher (placeholders + dashboard handoff). Modules are implemented in the orchestrator and execute as software workflows; success still depends on your approvals, API health, and tier limits. The builder runs multi-pass post-deploy checks (repo paths, Vercel READY, strict live URL + routes) and an artifact quality gate on key docs (placeholders, template-leak patterns, niche grounding) before calling a deploy “ready.” The U.S. government contracting planning pack is opt-in in the builder and is informational only (verify on SAM, USASpending, SBA, Census NAICS).

Technical verification endpoint: /api/provision?action=inventory.

Status legend

  • Live — In active use for paying builds, subject to tier, keys, and API health.
  • Partial — Works in many cases but has known provider/key dependencies or reduced fallback behavior.
  • Codegen — Client-side generation + repo files; no external API required.
  • Planned — Designed and specified; ships only when your paid tier's TIER_MODULES allowlist includes that module and vendor keys are configured (most wide integrations require Custom Volume).
